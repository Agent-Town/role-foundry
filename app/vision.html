Vision & System Overview

Role Foundry is a framework for training AI apprentices under honest, holdout-aware evaluation. This page explains the general system and how the current demo fits.

The framework

Role Foundry trains roles, not just models

Role Foundry is designed to train different roles over time. A role is a named apprentice with a public curriculum, sealed holdout evaluations, named teachers, and visible score deltas. The framework handles the training loop; the role defines what the apprentice learns.

What every role gets

  • A public curriculum the apprentice can practice on
  • Sealed holdouts the apprentice cannot see, for honest evaluation
  • Named teachers who score runs and promote failure themes
  • Visible score deltas between runs, especially on holdouts
  • Proof bundles with receipts judges can inspect
  • A failure-to-curriculum loop that improves teaching without leaking the exam

What varies per role

  • The curriculum content — product slices, code review, regression, etc.
  • The holdout episodes — different evaluation challenges per role
  • The teacher team — who authors and scores
  • The execution backends — which models build vs judge
  • The difficulty progression — how fast the apprentice levels up
Current concrete example

Software Engineer apprentice improving Role Foundry itself

The current alpha demo ships one concrete role: a Software Engineer apprentice that implements Role Foundry product slices under hidden-eval review. Robin + Neo are the teachers. The apprentice builds the system it is being trained by.

Honest scope note: The currently shipped curriculum slices are frontend/product-heavy because that is what the alpha app exposes. The "Software Engineer" framing reflects the intended breadth of the role — code review, regression prevention, documentation honesty — not a claim that all those curriculum families are shipped today.

System architecture

Training loop

  1. Teachers define a role — what good work looks like for this apprentice
  2. Public curriculum is published — scenarios the apprentice can train on
  3. Sealed holdouts stay teacher-only — the exam the apprentice cannot see
  4. The apprentice ships work — a slice, a fix, an artifact
  5. Teachers score the run — per-scenario notes, aggregate score, holdout deltas
  6. Failures become curriculum — themes are promoted, hidden prompts stay sealed

Key separations

  • Student vs teacher access: the apprentice sees public curriculum only; teachers see sealed holdouts and scoring internals
  • Public vs holdout evaluation: passing public curriculum is practice; passing sealed holdouts is the real signal
  • Demo vs live mode: demo ships pre-baked data for judges; live mode consumes configured exports from the runner bridge
  • Framework vs role: the training loop is general; the curriculum content is role-specific

Curriculum extension paths

Public curriculum families shipped

The public benchmark pack ships 12 episodes across 6 families derived from the current apprentice scenarios. These are all student-visible and safe to train on.

Sources: landing-story clarity, curriculum/holdout separation, score-delta legibility, proof-bundle visibility, demo honesty, failure-to-curriculum promotion.

Teacher-only holdout extension teacher-only

Teachers can extend the holdout lane with manually curated episodes from external sources like SWE-bench, Playwright docs, or code-review guides. These stay local-only and never enter the public repo or student-visible curriculum.

See docs/swe-bench-holdout-extension.md for the process and constraints. The extension path is intentionally small and manually curated.

Future roles (not shipped yet)

Role Foundry is built so that new roles can be added over time. Examples that could use the same framework:

  • Code Reviewer — trained on diff critique, scope discipline, uncertainty honesty
  • QA Engineer — trained on regression detection, test coverage, edge-case discovery
  • Documentation Writer — trained on artifact honesty, audience clarity, change traceability
  • DevOps Apprentice — trained on deployment safety, config validation, incident response

These are illustrative, not committed. The current demo ships only the Software Engineer role.

Why this distinction matters

Most AI training demos blur the line between "the framework can do anything" and "here is what we actually built." Role Foundry keeps it explicit:

  • The framework is general: any role can use the training loop, holdout separation, scoring, and proof bundles.
  • The current example is specific: one Software Engineer apprentice, two teachers, frontend/product-heavy curriculum, honest about what is shipped vs planned.
  • The proof is in the receipts: judges can see exactly what the apprentice did, how it scored, and what improved.
